# 0006 — Use Auth.js v5 credentials auth with JWT sessions

- **Status:** Accepted
- **Date:** 2026-07-17

## Context

M1 needs staff sign-in so shop surfaces can be protected. Constraints: agents must be able to
develop and e2e-test auth with zero external services (no vendor dashboards, no callbacks into
sandboxes), it must run on Next 16 (App Router + `proxy.ts`, since `middleware` is deprecated),
and it must not preclude customer accounts later. A sibling project (sybaris) runs this exact
stack on Next 16 in production — the pattern below is lifted from its working implementation.

## Decision

- **Auth.js v5** (`next-auth@beta`) with a **Credentials provider**: email + password against
  our own `user_accounts` table, hashed with **bcryptjs**, input parsed with **zod**.
- **JWT sessions** (no session table), with the person's id and staff roles embedded in the
  token.
- **Edge-safe config split**: `src/lib/auth.config.ts` holds session/callback/authorization
  logic with no database imports; `src/proxy.ts` builds a NextAuth instance from it for edge
  route-gating; `src/lib/auth.ts` spreads it and adds the DB-backed Credentials provider (node
  runtime only).
- **Accounts attach to people**: `user_accounts.person_id` → `people` (1:1). Roles keep living
  on `person_roles` — the account is a login method, not an identity. Sign-in currently
  requires at least one staff role; customer sign-in arrives with the milestone that needs it.
- Route protection is two-layered: `proxy.ts` gates `/shop/**` at the edge, and server code
  re-checks via `src/lib/authz.ts` (`requireStaff()`). The proxy is the outer wall, never the
  only one.
- `AUTH_SECRET` falls back to a fixed dev value outside production so `pnpm dev`/`pnpm e2e`
  stay zero-setup; production deployment must set it (fails loudly otherwise).

## Alternatives considered

- **Clerk** — polished, but a paid vendor account, network-dependent dev, and lock-in; agents
  can't self-serve in sandboxes.
- **Better Auth** — capable and TS-native, but no in-ecosystem precedent on Next 16; Auth.js has
  a proven sibling implementation to crib from.
- **Hand-rolled sessions** — auth is exactly where boring, audited plumbing beats bespoke code.

## Consequences

- Zero-setup auth in dev, tests, and CI; deterministic seeded staff logins for e2e.
- Credentials-only means password reset/invite flows are ours to build when shops onboard
  (accepted; M7 territory).
- JWT sessions mean role changes take effect on next sign-in, not instantly — fine at dive-shop
  scale; revisit with a session table if instant revocation ever matters.
- New runtime deps: `next-auth`, `bcryptjs`, `zod` (zod also becomes the standard input-parsing
  library for server actions going forward).
